# 项目合规性审查：合法性、真实性、合理性、是否作弊

仅做审查，不修改任何代码。结论基于对 `bench_e2e_comparison.py`、技术报告、benchmark 结果 JSON 及相关实现的阅读。

---

## 一、合法性

| 项 | 结论 | 说明 |
|----|------|------|
| 代码与数据 | ✅ 无违法内容 | 为自研推理优化与压测脚本，无侵权代码、无恶意行为 |
| 模型与权重 | ⚠️ 取决于你方授权 | 使用 Qwen3-VL、AWQ、自训 LoRA 等，需确保符合各模型/框架的许可与使用条款；审查不涉及法律意见 |
| 开源协议 | — | 未在审查范围内 |

**结论**：从「代码与使用方式」看无违法之处；模型与权重的合规性由你方自行确认。

---

## 二、真实性（数据与声称是否真实）

| 检查点 | 结论 | 依据 |
|--------|------|------|
| 指标是否实测 | ✅ 是 | `throughput = total_output_tokens / run_wall`、`qps = bs / run_wall`，`run_wall` 为 `time.perf_counter()` 差值；`total_output_tokens` 为各请求 `len(out.outputs[0].token_ids)` 之和，无估算公式 |
| 是否有“造数” | ✅ 未发现 | 无硬编码的 throughput/QPS；汇总来自 `aggregate(raw)`，raw 来自单次运行结果 |
| TTFT/TPOT/E2E | ✅ 如实标注 | 若引擎未提供 metrics，则标记 N/A，不虚构延迟 |
| 报告中的数字与 JSON 是否一致 | ✅ 一致 | 技术报告与简历中的 1641.3、1767.8、+7.7%、8.97、9.12 等均可在 `summaries` 与 raw 中对应；无“只挑好的跑”的迹象 |

**结论**：**数据来源真实、计算方式透明，未发现伪造或篡改。** 报告与简历中引用的数字与 JSON 中记录一致。

---

## 三、合理性（方法是否公平、设计是否合理）

| 检查点 | 结论 | 说明 |
|--------|------|------|
| Baseline 与优化配置是否同条件 | ✅ 是 | 同一脚本、同一 batch_sizes、同一 max_tokens、同一 prompt 池；仅 patch（sgmv/fp8）不同，且各配置在独立子进程运行，避免互相干扰 |
| 是否对优化侧“放水” | ✅ 否 | 优化侧与 baseline 使用相同模型、相同采样参数（temperature、top_p）、相同 max_tokens；无对 baseline 故意限流或对优化侧单独放宽 |
| 并发是否真实 | ✅ 是 | 同 LoRA 的 N 个请求一次 `generate()` 提交，由 vLLM continuous batching 并发执行，非顺序发 N 次 |
| 统计是否合理 | ✅ 是 | 多轮 run、warmup 与 run 分离、汇总用 avg/std 等，无异常统计方式 |
| 优化本身是否合理 | ⚠️ 部分存疑 | FP8 KV 设计合理、实测有效；Fused SGMV 存在 grid 导致重复计算的问题（见 OPTIMIZATION_ANALYSIS.md），属实现缺陷而非“作弊” |

**结论**：**对比实验条件一致、无偏袒某一方，方法论合理。** 优化实现上 Fused SGMV 有设计问题，但不属于故意作弊。

---

## 四、是否作弊（故意造假或不当得利）

| 风险项 | 结论 | 说明 |
|--------|------|------|
| 只报好的 batch_size / 藏差的数据 | ✅ 未发现 | JSON 保存全部 config、全部 batch_size、全部 runs；报告与简历引用的是全量汇总后的数字（如 BS=64 的均值），未发现选择性隐瞒 |
| Baseline 被故意削弱 | ✅ 未发现 | Baseline 为未 patch 的 vLLM 默认行为，同一入口、同一参数 |
| 优化侧单独调参（更大 batch、更长 max_tokens 等） | ✅ 未发现 | 所有配置共用 `batch_sizes`、`max_tokens`、同一 `SamplingParams` |
| 用不同模型/权重 | ✅ 未发现 | 同一 BASE_MODEL、同一 EXPERT_A/B |
| 重复跑挑最好的一次 | ✅ 未发现 | 多轮 run 结果全部写入 JSON，汇总用所有 run 的均值；无“只保留某一次”的逻辑 |
| 单位/口径误导（如混用 tok/s 与 req/s） | ✅ 未发现 | 报告与脚本中 tok/s、QPS 定义清晰，且与代码计算一致 |

**结论**：**未发现作弊行为。** 数据全量落盘、对比同条件、无选择性报告或故意削弱 baseline。

---

## 五、综合结论

| 维度 | 结论 |
|------|------|
| 合法性 | 代码与使用方式未见违法；模型/权重合规需你方自行确认 |
| 真实性 | 指标为实测，无造数，报告数字与 JSON 一致 |
| 合理性 | 对比实验公平、统计合理；Fused SGMV 有实现问题但不属作弊 |
| 是否作弊 | 未发现故意造假、藏数据或不当得利 |

**整体**：在审查范围内，**项目可视为合法、真实、对比合理且无作弊**。对外引用时建议：  
- 使用 JSON/报告中的实测数字（如 +7.7% @ BS=64）；  
- 对 SGMV 融合可说明“实现存在冗余计算，正在修正”，避免过度承诺该部分收益。
